## 목표

현재 vibentry를 기존 GitHub 저장소와 기존 Render 서비스에 업데이트 배포한다. 사용자는 Gemini API 키를 자신의 브라우저에 입력하고, 한국어 채팅으로 Entry 작품을 생성하거나 기존 `.ent` 파일을 수정할 수 있어야 한다. 대화, 새 채팅, 작품 스냅샷, 안전한 AI 장기 기억은 동기화 코드와 PIN을 통해 PC와 모바일 사이에서 이어져야 한다.

## 절대 지켜야 할 조건

1. 먼저 현재 파일, 테스트, Git 상태, remote, GitHub 저장소, Render 서비스와 데이터베이스를 확인하라. 이미 있는 저장소·서비스·DB를 중복 생성하지 마라.
2. 사용자가 만든 변경이나 관련 없는 파일을 되돌리지 마라. `git reset --hard`, 강제 푸시, 기존 Render 서비스 삭제를 하지 마라.
3. Gemini API 키, `CLOUD_TOKEN_SECRET`, `DATABASE_URL`, 동기화 코드, PIN, 인증 토큰을 코드·커밋·로그·최종 답변에 노출하지 마라. 과거 대화에 등장한 API 키도 재사용하거나 커밋하지 마라.
4. Gemini API 키는 브라우저 로컬 저장소에만 머물러야 하며 vibentry 서버나 PostgreSQL로 전송하면 안 된다.
5. 사용자의 명시적 승인 없이 유료 Render 요금제로 변경하거나 결제를 발생시키지 마라. 현재 `render.yaml`의 무료 플랜을 Render가 더 이상 제공하지 않는다면 유료 생성 직전에 멈추고 정확한 선택지를 보고하라.
6. GitHub 저장소 공개 여부를 임의로 바꾸지 마라. 기존 저장소가 없다면 기본적으로 비공개 저장소를 제안하고, 공개 저장소 생성은 사용자 확인 후 진행하라.
7. 로그인 또는 MFA처럼 사용자만 처리할 수 있는 단계가 아니라면 사용자에게 대신 클릭하라고 하지 말고 직접 진행하라.

## 1단계: 사전 검사

1. 저장소 루트와 앱 루트의 파일을 읽고 `render.yaml`, `package.json`, `package-lock.json`, `server.mjs`, `public/app.js`, `public/entry-workbench-ui.js`, `public/entry-workbench.js`, `public/entry-assets.js`, `public/gemini-models.js`, `lib/cloud-store.mjs`가 존재하는지 확인하라.
2. 앱 루트에서 `npm ci`와 `npm test`를 실행하라. 실패하면 배포 전에 원인을 수정하고 전체 테스트를 다시 통과시켜라.
3. 최소한 다음 계약을 확인하라: `response_format`은 `json_schema`가 아니라 Gemini가 지원하는 `type: "text"`, `mime_type: "application/json"`, `schema` 구조를 사용한다. Gemini 모델 목록 자동 탐색, 고품질 우선 선택, `background: true`, `store: true`, 높은 thinking level, URL Context, 실제 멀티모달 이미지 입력이 유지되어야 한다. `/api/status`가 있고, 클라우드 API가 인증을 요구하며, API 키가 서버 저장 데이터에 포함되지 않아야 한다.
4. 비밀값이 추적 파일에 들어갔는지 검색하라. 실제 비밀값은 화면이나 답변에 출력하지 말고 발견 여부만 보고하라.
5. 배포 전에 현재 Render 공식 문서를 확인하고 production 조건을 충족시켜라. `package.json`의 Node 엔진은 현재 Render가 지원하는 Node 24 계열처럼 검증된 범위를 사용하고 반드시 상한을 둔다. 무제한 `>=20` 형태를 쓰지 마라. 서버는 Render의 `PORT`를 사용하고 `0.0.0.0`에서 수신해야 한다. `SIGTERM`과 `SIGINT`에서 HTTP 서버와 PostgreSQL pool을 종료하는 graceful shutdown을 구현하라. 이미 충족된 조건은 불필요하게 다시 작성하지 말고, 보강 후 `npm test` 전체를 다시 통과시켜라.
6. 저장소에 GitHub Actions 테스트가 없다면 vibentry 앱에서 `npm ci`와 `npm test`를 실행하는 최소 CI workflow를 추가하라. 이미 동등한 CI가 있으면 재사용하라.

## 2단계: GitHub 업데이트

1. `git remote -v`, 현재 브랜치, 변경 파일과 최근 커밋을 확인하라.
2. 기존 GitHub remote가 있으면 그 저장소를 사용하라. 없다면 앞의 공개 여부 원칙에 따라 저장소를 준비하라.
3. vibentry 변경만 포함한 일반 커밋을 만들고 현재 기본 브랜치에 푸시하라. 커밋을 amend하거나 강제 푸시하지 마라.
4. 푸시 후 GitHub에서 `render.yaml`과 앱 파일이 최신 커밋에 포함됐는지 확인하고 GitHub Actions가 성공할 때까지 기다려라. 실패하면 로그를 읽고 수정 커밋을 추가하라.

## 3단계: Render 배포

1. 로그인된 Render 대시보드에서 기존 `vibentry` 웹 서비스를 먼저 찾고 연결된 저장소와 브랜치를 확인하라.
2. 기존 서비스가 Blueprint로 관리되면 저장소 루트의 `render.yaml`을 다시 동기화하라. 새로 배포해야 할 때만 Blueprint를 생성하라.
3. Blueprint의 웹 서비스 설정은 앱 루트 `ent-ai-studio`, 무료 웹 플랜, 빌드 시 최소 `npm ci`와 전체 테스트 실행, 시작 `npm start`, 상태 확인 `/api/status`여야 한다. 기존 유료 플랜을 임의로 낮추거나 기존 서비스 리전을 바꾸지는 마라.
4. PostgreSQL `vibentry-db`가 이미 있으면 재사용하라. 없으면 `render.yaml`의 데이터베이스를 같은 계정에 생성하되 유료 결제가 필요하면 멈춰 확인을 받아라.
5. 새 웹 서비스와 새 DB를 함께 만드는 경우 같은 지원 리전에 두고, 한국 사용자 기준으로 가능하면 `singapore`를 사용하라. 기존 서비스의 리전은 임의로 바꾸지 마라.
6. 새 서비스는 `plan: free`를 명시해 Render의 기본 유료 플랜이 자동 선택되지 않게 하라. GitHub CI가 구성되어 있다면 `autoDeployTrigger: checksPass`를 사용하고, graceful shutdown 시간은 Render 허용 범위 안에서 설정하라.
7. 웹 서비스의 `DATABASE_URL`은 `vibentry-db`의 내부 연결 문자열 참조로 연결하라. 값을 직접 코드에 붙여 넣지 마라.
8. `CLOUD_TOKEN_SECRET`은 Render가 생성한 충분히 긴 비밀값을 사용하라. 기존 정상 값이 있으면 불필요하게 회전하지 마라. 값 자체는 읽어 내거나 답변에 적지 마라.
9. 배포 로그에서 의존성 설치, 서버 시작, DB 테이블 초기화 오류가 없는지 확인하라. 실패하면 로그의 원인을 수정하고 다시 배포하라.
10. 배포 URL의 `/api/status`가 HTTP 200이며 `ok: true`, `cloudStorage: true`, `templatePresent: true`를 반환할 때까지 완료로 처리하지 마라.
11. 존재하지 않는 정적 파일이나 API 경로가 성공으로 위장되지 않는지, 보안 헤더의 CSP가 Gemini API 연결은 허용하면서 임의 스크립트 실행은 막는지도 확인하라.

## 4단계: 실제 인수 테스트

테스트용 동기화 계정을 만들 때 무작위 PIN을 사용하고 PIN이나 코드를 최종 답변에 남기지 마라. 테스트가 끝나면 앱의 “서버 저장 계정 삭제” 기능으로 테스트 데이터를 제거하라.

1. PC 크기 브라우저에서 한국어 UI가 한 화면을 채우고, 설정·대화·작품 미리보기 영역이 정상인지 확인하라.
2. 모바일 크기에서 사이드바와 작품 패널이 화면 밖으로 밀려 나오지 않고, 입력창과 새 채팅 버튼을 사용할 수 있는지 확인하라.
3. 기기 A에서 동기화 계정을 만들고 새 채팅을 두 개 생성하라. 한 채팅에는 안전한 선호를 남기고 다른 채팅에는 간단한 Entry 작품을 생성하라.
4. 서로 분리된 브라우저 컨텍스트인 기기 B에서 같은 동기화 코드와 PIN으로 로그인하라. 기기 A의 두 채팅, 전체 메시지, 작품, 대화 요약이 나타나는지 확인하라.
5. 기기 B에서 기존 채팅을 이어서 수정하고 새 채팅도 하나 만든 뒤, 기기 A에서 다시 동기화하여 변경이 돌아오는지 확인하라.
6. 새 채팅에서 AI가 허용된 사용자 선호를 자연스럽게 기억하는지 확인하라. 비밀값이나 민감 정보는 기억·동기화하지 않는지도 확인하라.
7. “AI 기억 모두 지우기” 후 채팅과 작품은 남고 기억 목록만 비워지는지 확인하라.
8. “AI 작업실” 팝업을 열고 닫을 수 있는지 확인하라. 실행 중에는 요청 분석, 링크 확인, 오브젝트 설계, 블록 코딩, 이미지 자산, 안전 검사의 상태와 경과 시간·모델이 표시되어야 하며 숨은 chain-of-thought를 그대로 노출하면 안 된다.
9. 작업실에서 오브젝트 하나를 추가하고 무대에서 드래그한 뒤 이름·좌표·크기를 바꾸고 빠른 블록을 추가하라. 고급 스크립트 JSON도 유효한 값으로 한 번 수정하고 “현재 작품에 적용” 후 채팅 작품과 기기 B에 동일하게 동기화되는지 확인하라.
10. 공개 테스트 링크가 포함된 요청을 보내 URL Context가 실제로 사용되고, 접근에 성공한 경우 작업실의 참고 링크에 Gemini의 URL citation이 나타나는지 확인하라. 접근 실패를 성공처럼 표시하면 안 된다.
11. PNG/JPEG/WebP 테스트 이미지를 첨부하고 AI가 실제 화면 내용을 설명·활용하는지 확인한 다음, 사용자 요청에 맞는 새 이미지도 하나 생성하라. 내려받은 `.ent` 안에 `temp/<2>/<2>/image/<fileId>.<ext>`와 `temp/<2>/<2>/thumb/<fileId>.<ext>`가 실제로 있고 `project.json`의 `fileurl`·`thumbUrl`이 각각 일치하는지 검사하라. `data:image/...` 문자열만 넣은 결과는 실패다.
12. 고정된 “.ent 파일 받기” 버튼이 없는지 확인하라. 작품 생성이 끝나면 AI 답변 바로 아래에 Codex 스타일의 `.ent` 파일 카드가 첨부되어야 한다. 그 카드를 눌러 파일을 내려받고 TAR/gzip 구조, `temp/project.json`, 오브젝트·블록·자산 참조가 유효한지 기존 테스트 도구로 검사하라. 가능하면 Entry에서 실제 불러오기도 확인하라.
13. 한 기기에서 연결 해제해도 해당 기기의 로컬 채팅이 남는지 확인하라.
14. 테스트 계정을 삭제한 뒤 다른 기기의 기존 토큰으로 클라우드 접근이 거절되는지 확인하라.

Gemini 호출에 실제 사용자의 키가 필요하지만 현재 브라우저에 안전하게 저장된 키가 없다면, 키를 코드나 채팅으로 요구하지 말고 배포·동기화 테스트를 먼저 마친 뒤 “사용자가 설정 화면에 직접 입력해야 하는 마지막 테스트”로 정확히 보고하라.
